    Specifiche rendicontazione celiachia

    Prima revisione documento: 18 settembre 2025
    Ultima revisione documento: 14 ottobre 2025

    Introduzione

    Nell'ambito del progetto di integrazione del metodo di pagamento tramite tessera sanitaria per clienti affetti da celiachia, è stato sviluppato un meccanismo di raccolta e gestione dei dati utili ad effettuare la rendicontazione per tali metodi di pagamento.

    Il sistema ha l’obiettivo di:

    • consentire la registrazione puntuale delle transazioni di tipo Celiachia provenienti dai sistemi di cassa Posware;
    • rendere disponibili i dati in formato standardizzato e consultabile per i sistemi di BackOffice;
    • abilitare la successiva trasmissione dei dati al provider Celiachia/ASL di riferimento, che si occupa della riconciliazione e della rendicontazione ufficiale.

    Per soddisfare questi requisiti è stata introdotta la tabella celiac_evoucher_settlement, che funge da repository unico per tutte le transazioni Celiachia. Ogni record corrisponde a un documento commerciale di vendita emesso, in cui sono presenti una o più transazioni pagamento effettuate usando il bonus celiachia (eVoucher Celiachia), memorizzato insieme al relativo payload JSON (CeliacSettlementRecord), il quale descrive nel dettaglio i pagamenti e i prodotti autorizzati.

    Questa documentazione descrive la struttura della tabella, i campi che la compongono e il formato del JSON che viene memorizzato, così che si possa:

    • comprendere la logica di memorizzazione del dato,
    • leggere e interpretare correttamente i dump forniti (sia MySQL che SQL Server),
    • integrare i dati nei propri processi di rendicontazione.

    Specifiche tecniche

    La tabella celiac_evoucher_settlement raccoglie i dati relativi alla rendicontazione delle transazioni gestite tramite eVoucher Celiachia. Il dato principale viene memorizzato in formato JSON all’interno della colonna JsonData, che segue la struttura del modello CeliacSettlementRecord.

    Questa sezione descrive:

    • la struttura della tabella (sia MySQL che SQL Server),
    • il significato dei campi principali,
    • la struttura dettagliata del JSON memorizzato.

    Struttura tabella celiac_evoucher_settlement

    MySQL

    Query CREATE TABLE
    CREATE TABLE `celiac_evoucher_settlement` (
  `Date` date NOT NULL,
  `StoreId` int NOT NULL,
  `TerminalId` int NOT NULL,
  `TransactionId` int NOT NULL,
  `JsonData` json NOT NULL,
  `UpdatedAt` datetime(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6),
  `SequenceId` bigint NOT NULL AUTO_INCREMENT,
  PRIMARY KEY (`Date`,`StoreId`,`TerminalId`,`TransactionId`),
  UNIQUE KEY `IX_celiac_evoucher_settlement_SequenceId` (`SequenceId`),
  KEY `IX_celiac_evoucher_settlement_TerminalId` (`TerminalId`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;

    SQL Server

    Query CREATE TABLE
    CREATE TABLE posware.dbo.celiac_evoucher_settlement (
    Date date NOT NULL,
    StoreId int NOT NULL,
    TerminalId int NOT NULL,
    TransactionId int NOT NULL,
    JsonData nvarchar(MAX) COLLATE Latin1_General_CI_AS NOT NULL,
    UpdatedAt datetime2 NOT NULL,
    SequenceId bigint IDENTITY(1,1) NOT NULL,
    CONSTRAINT PK_celiac_evoucher_settlement PRIMARY KEY (Date,StoreId,TerminalId,TransactionId)
);

CREATE UNIQUE NONCLUSTERED INDEX IX_celiac_evoucher_settlement_SequenceId 
    ON posware.dbo.celiac_evoucher_settlement (SequenceId ASC);

CREATE NONCLUSTERED INDEX IX_celiac_evoucher_settlement_TerminalId 
    ON posware.dbo.celiac_evoucher_settlement (TerminalId ASC);

    Creazione tabella in Posware 4.2

    ATTENZIONE! SOLO PER POSWARE 4.2

    Con Posware 4.2, è necessario aggiornare manualmente il server di barriera per garantire la corretta integrazione con l'aggiornamento sul bonus celiachia.

    L'aggiornamento lato server di barriera deve essere eseguito tramite l'utilizzo degli script SQL forniti con l'installazione di Posware.
    Durante l’aggiornamento di Posware sulla cassa, il setup crea automaticamente una cartella denominata Modif all'interno della cartella di Posware (percorso predefinito di default: C:\Programmi\Posware\Modif).
    All’interno di questa cartella vengono generati due file SQL, contenenti le query necessarie per aggiornare la struttura del database del server di barriera:

    • celiac_evoucher_settlement_mysql.sql
    • celiac_evoucher_settlement_sqlserver.sql

    I tecnici devono eseguire manualmente lo script corrispondente al tipo di database in uso (MySQL o SQL Server) una sola volta direttamente sul database del server di barriera.

    Descrizione dei campi tabella celiac_evoucher_settlement

    Campo Tipo Descrizione
    Date date Data della transazione (corrisponde al giorno di emissione dello scontrino)
    StoreId int Identificativo univoco del punto vendita
    TerminalId int Identificativo univoco della cassa
    TransactionId int Identificativo univoco della transazione lato Posware
    JsonData json/nvarchar(MAX) Dato transazionale in formato JSON conforme al modello CeliacSettlementRecord
    UpdateAt datetime Timestamp di ultima modifica/inserimento del record
    SequenceId bigint Identificativo univoco incrementale interno (surrogate key tecnica)

    Chiave primaria e vincoli

    La chiave primaria è composta da (Date, StoreId, TerminalId, TransactionId).
    Il campo SequenceId garantisce un ordinamento cronologico e permette la paginazione ottimizzata nelle interrogazioni.

    Struttura del JSON CeliacSettlementRecord

    Il campo JsonData contiene un oggetto JSON con la seguente struttura:

    CeliacSettlementRecord
    {
  "ProviderTransactionId": "<Numero transazione attribuito dal provider>",
  "PoswareTransactionId": "<Numero transazione attribuito da Posware>",
  "TransactionDate": "<Data e ora di emissione dello scontrino>",
  "StoreId": <Codice punto vendita>,
  "TerminalId": <Codice della cassa>,
  "ReceiptNumber": <Numero dello scontrino>,
  "Payments": [
    {
      "AuthCode": "<Codice autorizzazione generato dal sistema Celiachia>",
      "IsVoided": <true/false>,
      "RequestedAmount": <Importo richiesto>,
      "AuthorizedAmount": <Importo autorizzato>,
      "Products": [
        {
          "Barcode": "<EAN/UPC del prodotto>",
          "Quantity": <Quantità erogata>,
          "Amount": <Importo totale per il prodotto>
        }
      ]
    }
  ],
  "Metadata": { ... } 
}

    Dettaglio campi JSON

    Livello principale (CeliacSettlementRecord)
    Campo Tipo Descrizione
    ProviderTransactionId string Identificativo transazione assegnato dal provider del sistema Celiachia
    PoswareTransactionId int Identificativo transazione lato Posware
    TransactionDate datetime Data e ora della transazione (emissione dello scontrino)
    StoreId int Codice identificativo del punto vendita
    TerminalId int Codice identificativo della cassa
    ReceiptNumber int Numero progressivo dello scontrino
    Payments array Collezione dei pagamenti associati alla transazione
    Metadata object Struttura opzionale per informazioni aggiuntive/estensioni future
    Livello Payments
    Campo Tipo Descrizione
    AuthCode string Codice autorizzazione rilasciato dal sistema Celiachia
    IsVoided bool Indica se il pagamento è stato successivamente stornato
    RequestedAmount decimal Importo totale richiesto al sistema Celiachia
    AuthorizedAmount decimal Importo effettivamente autorizzato dal sistema Celiachia
    Products array Lista dei prodotti coperti dal pagamento
    Livello Products
    Campo Tipo Descrizione
    Barcode string Codice a barre del prodotto (EAN, UPC, o altro identificativo univoco)
    Quantity decimal Quantità venduta
    Amount decimal Importo totale attribuito al prodotto

    Conclusione

    In conclusione:

    • La tabella celiac_evoucher_settlement funge da repository unico per le transazioni Celiachia.
    • I campi principali permettono query efficienti su chiavi logiche (Date, StoreId, TerminalId, TransactionId).
    • Il JSON CeliacSettlementRecord fornisce i dettagli granulari della transazione e può essere esteso in futuro tramite Metadata.
