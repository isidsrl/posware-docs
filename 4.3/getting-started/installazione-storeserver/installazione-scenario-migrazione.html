    Posware StoreServer

    Scenario di migrazione

    Prima revisione documento: 24 aprile 2024
    Ultima revisione documento: 24 luglio 2024

    Glossario

    • MVP: acronimo di Minimum Viable Product, ossia prodotto minimo funzionante, è la versione di un prodotto con caratteristiche appena sufficienti per essere utilizzabile dai primi clienti, i quali possono quindi fornire feedback per lo sviluppo futuro del prodotto stesso;
    • PPC: acronimo di Posware Price Checker, rappresenta uno dei moduli che compongono l'infrastruttura di Posware;
    • WebApp: applicazione web che offre la possibilità di gestire in modo centralizzato le funzionalità disponibili nello StoreServer;

    Cenni preliminari

    Il presente documento, come introdotto nell'overview generale, descrive nel dettaglio la procedura di installazione dello StoreServer in caso di migrazione da Posware 4.2.x a 4.3, dando particolare enfasi ai vari passaggi necessari per portare a compimento quest'operazione. Vengono inoltre fornite delle linee guida da seguire in caso si dovessero verificare delle problematiche durante tale procedura, arrivando nel caso peggiore anche allo scenario di rollback.

    Prerequisiti

    Prima di procedere con le diverse operazioni per installare lo StoreServer in caso di migrazione di Posware dalla versione 4.2.x alla 4.3, è assolutamente necessario aver installato i vari applicativi indicati nei requisiti minimi dell'overview generale ed essersi assicurati di avere le risorse hardware minime indispensabili.

    Inoltre, è fondamentale aver eseguito quelle procedure comuni ai due scenari di installazione che vanno completate prima di procedere con l'installazione effettiva. In particolare, controllare ancora una volta di:

    • avere dei backup integri della situazione precedente all'installazione dello StoreServer, soprattutto per quanto riguarda il database;
    • aver aggiornato e configurato adeguatamente il provider del database del server centrale, da cui bisognerà poi effettuare la migrazione tramite il Database Upgrader. A tal proposito si ricorda ancora una volta che per MySQL la versione compatibile con lo StoreServer è necessariamente la 8.4, mentre per SQL Server è possibile scegliere tra SQL Server 2016, 2017, 2019 o 2022.

    Infine, prima di iniziare i vari passaggi necessari all'installazione dello StoreServer, assicurarsi che le casse che si vogliono aggiornare a Posware v. 4.3 abbiano già inviato tutti i dati di vendita al server di barriera e, in caso affermativo, procedere poi a spegnere tali casse fino allo step in cui sarà necessario associarle allo StoreServer.

    Warning

    Al termine di questa guida è previsto che venga effettuato un invio massivo di tutta l'anagrafica del punto vendita allo StoreServer. Prima di iniziare, assicurarsi di poter effettuare questa operazione con il punto vendita senza causare effetti collaterali indesiderati.

    Operazioni necessarie

    Al fine di migrare Posware 4.2 allo StoreServer 4.3 sarà necessario eseguire tutti questi step:

    1. Migrazione del database tramite il Database Upgrader
    2. Aggiornamento dei servizi legacy
    3. Installazione StoreServer
    4. Inserire tutta la directory dello StoreServer nella lista delle esclusioni di antivirus e firewall di sistema
    5. Avviare lo StoreServer
    6. Associare le casse allo StoreServer
    7. Verificare la ricezione dei log e dei dati delle statistiche dettagliate
    8. Verificare l'allineamento delle tabelle log e transactionslogextra del database [valutare con Alberto se va lasciato]
    9. Verificare il comportamento corretto dei servizi legacy
    10. Inviare l'intera anagrafica per ripopolare le tabelle dell'ex database CASSAMASTER

    Installazione

    Migrazione del database tramite il Database Upgrader

    Il primo step dell'installazione dello StoreServer consiste nella migrazione del database utilizzato in Posware 4.2, così da renderlo compatibile con la nuova architettura. Per fare ciò è necessario utilizzare il Database Upgrader, un software sviluppato appositamente per questo scopo, che viene fornito assieme allo StoreServer.

    Prima di procedere, però, bisogna assicurarsi che nel file connectionStrings.Production situato all'interno della directory con tutti i vari file del Database Upgrader sia impostata la Connection string al database, MySQL o SQL Server, che veniva utilizzato in Posware 4.2, ossia quello che deve essere migrato. Oltre a ciò, affinché il database alla fine della migrazione abbia il nome opportuno, va rinominato il database del cliente in "posware42". Infine, configurare il file databaseUpgraderSettings adeguatamente in base alla quantità di dati che si desidera migrare nello StoreServer, tenendo in considerazione che sia il campo MonthsToImport che MonthsToImportForRistampaScontrini comprendono nel conteggio anche il mese attuale.

    Una volta impostati adeguatamente tutti questi file e il nome del database, procedere con il lancio del Database Upgrader. Se tutto è stato configurato correttamente, il Database Upgrader dovrebbe partire senza problemi e mostrare la seguente schermata (nel caso il database utilizzato sia MySQL):

    A questo punto premendo il pulsante centrale nella schermata verrà creato il nuovo database su cui verrà poi in seguito migrato il vecchio e, dopo una rapida analisi delle varie tabelle per impostare quali importare, si potrà avviare la migrazione premendo il relativo pulsante.

    Una volta completata la migrazione, l'utente verrà notificato dall'applicazione e potrà procedere con l'aggiornamento dei servizi legacy se tutto è andato a buon fine oppure con la consultazione del log del Database Upgrader in caso di warning o errori riscontrati nella procedura di migrazione.

    NOTA BENE: Se il database migrato è un database MySQL, il Database Upgrader potrebbe assegnargli un nome casuale al posto di "posware". In tal caso è assolutamente necessario rinominarlo prima di proseguire con gli step successivi della guida. Visto che MySQL attualmente non permette di rinominare i database, è consigliabile eseguire un dump del database e rimportarlo assegnandogli il nome "posware".

    Troubleshooting per i tecnici

    Per qualsiasi problema possa sorgere con il Database Upgrader, la soluzione primaria è sempre quella di consultare il rispettivo log. Il file di log si trova dentro la directory del Database Upgrader e il suo nome ha la seguente struttura: dbupgrader-log_YYYY-MM-DD.log.

    Vengono riportati di seguito alcuni dei casi d'uso più frequenti in cui possono avvenire degli errori o dei warning con il Database Upgrader.

    ERRORE: Impossibile rilevare la versione del database

    Questo errore avviene quando il Database Upgrader non rileva il database registrato nella Connection string, quindi controllare innanzitutto che tutti i parametri della Connection string siano giusti e, se lo sono, assicurarsi che il database sia attualmente raggiungibile.

    WARNING: I campi fuori standard non sono stati migrati

    Questo warning compare se il Database Upgrader trova nel database sorgente delle colonne in più rispetto a quelle che dovrebbero esserci nel database di destinazione. Questo può accadere per due motivi:

    • La colonna custom era stata aggiunta da qualcuno e quindi il Database Upgrader rilevandola come tale non la importerà.
    • La colonna custom era presente in versioni precedenti di Posware, ma poi si è deciso di eliminarla nelle nuove versioni e quindi il Database Upgrader non la migra.

    In ogni caso, quando si incontra questo messaggio bisogna sempre andare a guardare nel log quali sono le colonne che sono state considerate custom per assicurarsi che non sia stata considerata come tale una colonna fondamentale. Se ciò dovesse succedere, bisogna aprire una segnalazione per fare includere tali colonne successivamente nella migrazione e sistemare il comportamento del Database Upgrader.

    WARNING: Alcuni record duplicati o con errori non sono stati migrati

    Questo warning viene notificato se il Database Upgrader trova dei record duplicati o in cui sono presenti degli errori durante la migrazione dal database sorgente a quello di destinazione.

    Una delle casistiche più comuni in cui può accadere questa situazione si ha quando la tabella in cui sono stati rilevati i record duplicati era, in origine, una tabella senza chiavi primarie a cui sono state successivamente aggiunte con la migrazione alla nuova versione di Posware. Una delle tabelle in cui è riscontrabile frequentemente questo problema è la tabella isi_menu.

    Quando si incontra questo messaggio va assolutamente consultato il log per vedere quali sono i record che non sono stati migrati per vedere se c'è la possibilità di migrarli manualmente in caso di errore. Mentre, in presenza di record duplicati, bisogna controllare se quello eliminato era effettivamente quello errato tra i due duplicati, perché, in caso contrario, bisogna procedere a eliminare dalla tabella il record errato sostituendolo con il duplicato corretto.

    Aggiornamento dei servizi legacy

    Prima di passare all'installazione effettiva dello StoreServer, è necessario completare l'aggiornamento dei servizi legacy. I dettagli specifici sono consultabili nella rispettiva sezione.

    Installazione StoreServer

    Per installare lo StoreServer portare la cartella relativa nel computer del cliente nel percorso "C:\Users\Utente\AppData\Local\isid", accertandosi che siano presenti nella cartella Modules della directory dello StoreServer anche le cartelle dei moduli necessari.

    Per la prima release MVP servono soltanto il PoswareModule e il PPC (Posware Price Checker).

    Eliminare, se presenti nella directory dello StoreServer, i file appsettings.Development.json e appsettings.Staging.json. Impostare adeguatamente l'InstanceConfigurationId e la Connection string al database nuovo (MySQL o SQL Server), quello creato con la migrazione tramite il Database Upgrader, nel file appsettings.Production.json, in modo che rispecchi la configurazione del cliente. In particolare, l'InstanceConfigurationId deve essere pari all'identificativo del punto di vendita considerato. Inoltre, configurare il parametro OnDuplicateLogStrategy con valore "Ignore".

    Infine, installare lo StoreServer come servizio di Windows e procedere a configurare alcune sue proprietà. In particolare:

    • In "Generale" impostare il "Tipo di avvio:" pari a "Automatico".
    • In "Connessione" selezionare la voce "Account di sistema locale".
    • In "Ripristino" assegnare il valore "Riavvia il servizio" a "Primo tentativo:", "Secondo tentativo:" e "Tentativi successivi:" e selezionare la checkbox "Attiva azioni per arresto con errori".

    Antivirus e firewall

    Dopo l'installazione, è necessario inserire l'intera directory dello StoreServer nella lista delle esclusioni sia dell'antivirus che del firewall di sistema, se presenti, prima di provare ad avviarlo.

    Avvio dello StoreServer

    Arrivati a questo step, avviare lo StoreServer lanciando il file Posware.Services.Dmo.exe nella directory del programma. Fatto ciò si aprirà il prompt dei comandi con alcuni messaggi di info tra cui la root con cui potersi collegare alla WebApp.

    Una volta collegatosi allo StoreServer, ci si ritroverà nella Dashboard generale, che rappresenta la home page della WebApp, in cui sono presenti alcuni componenti che dopo aver associato le casse all'applicazione mostreranno alcune informazioni generali, come l'incasso giornaliero e quello settimanale o lo stato delle casse collegate.

    Troubleshooting per i tecnici

    Per qualsiasi problema possa sorgere con lo StoreServer, la soluzione primaria è sempre quella di consultare i rispettivi log. I file di log sono tre e si trovano dentro la directory dello StoreServer.
    Illustriamo di seguito la struttura dei loro nomi e le informazioni riportate in ognuno di essi:

    • dmo-ef-log_YYYY-MM-DD.log: questo log contiene tutte le informazioni legate a Entity Framework (EF) Core, quali le query tradotte e lanciate dal framework e le informazioni riguardanti la gestione delle connessioni al database.
    • dmo-log_YYYY-MM-DD.log: questo è il log centrale dello StoreServer e contiene le informazioni di base, i vari warning ed errori che si verificano nell'applicazione, oltre alle informazioni che sono state inserite volutamente dagli sviluppatori, affinché rimangano tracciate. Non contiene i log di EF Core e delle richieste del server web, ASP.NET Core.
    • dmo-overall-log_YYYY-MM-DD.log: questo è il log completo che contiene qualsiasi informazione venga fornita dallo StoreServer, comprese anche quindi tutte le informazioni contenute negli altri due log.

    Vengono riportati di seguito alcune delle problematiche che potrebbero emergere durante il primo avvio dello StoreServer.

    No connection string found in appsettings.json

    Questo errore si verifica quando non è stata impostata nessuna Connection string al database nei file appsettings.json e appsettings.Production.json. Quindi assicurarsi di compilare il relativo campo e procedere nuovamente a lanciare lo StoreServer.

    Errore generico nella Dashboard generale della WebApp

    In caso si verifichino degli errori una volta collegatosi allo StoreServer comparirà nella Dashboard generale della WebApp un messaggio a schermo generico di errore che indica l'impossibilità di ricevere i dati, con al di sotto di esso un pulsante per provare a ricaricare la schermata.

    Prima di premere il pulsante Riprova, si consiglia di effettuare le seguenti verifiche per cercare di individuare la fonte che ha causato l'errore durante la richiesta delle informazioni principali della Dashboard generale.

    Verifica della connettività lato server

    Assicurarsi che lo StoreServer sia operativo e perfettamente funzionante, per far sì che le chiamate arrivino a destinazione.

    Verifica della connettività lato client in caso di dispositivo remoto

    In caso la WebApp sia installata su un dispositivo remoto, ossia un dispositivo diverso da quello in cui è installato lo StoreServer, assicurarsi che la connessione ad Internet sia operativa e stabile. Controllare, inoltre, che il dispositivo remoto sia collegato sulla stessa rete logica dello StoreServer.

    Verifica della connettività del database

    Se le precedenti verifiche sono andate a buon fine, provare a dirigersi tramite la side bar sinistra della WebApp su Stato servizi, raggiungibile tramite la relativa icona, e controllare se la voce Database risulta essere online. In caso negativo, assicurarsi che sia il provider sia la Connection string siano corretti.

    Individuare eventuali messaggi di errore

    Per individuare eventuali messaggi di errore, si possono provare due approcci distinti:

    • Consultare dal browser che si sta utilizzando la voce Rete (Network) degli Strumenti per sviluppatori

    • Controllare il log dello StoreServer alla ricerca di una riga in cui sia presente la dicitura ERROR

    Associazione delle casse allo StoreServer

    Per associare le casse allo StoreServer, innanzitutto bisogna andare nella sezione apposita della WebApp e per fare ciò cliccare sulla voce Dashboard, sotto alla voce Casse della sezione Dispositivi della side bar sinistra dell'applicazione.

    In assenza di casse installate si potrà procedere ad associare una cassa premendo il relativo pulsante al centro della schermata.

    Fatto ciò, si aprirà una finestra pop-up in cui inserire l'indirizzo IP della cassa fisicamente presente in negozio che si vuole associare allo StoreServer, così che possa iniziare a trasmettere dati e venduti, oltre a ricevere le variazioni.

    Se la cassa è attualmente raggiungibile sull'indirizzo IP inserito ed è avviata ferma sulla schermata di login, allora la finestra pop-up si chiuderà, uscirà un messaggio in alto a destra di conferma sull'avvenuta associazione, che rimarrà visibile qualche secondo prima di svanire, e la cassa comparirà nell'elenco con le sue informazioni fondamentali.

    Troubleshooting per i tecnici

    Se durante l'associazione della cassa allo StoreServer dovesse uscire l'errore mostrato nell'immagine sottostante, controllare che la cassa sia raggiungibile all'indirizzo IP specificato e che sia online ed avviata sulla schermata di login.

    In ogni caso la soluzione primaria rimane sempre quella di consultare i log dello StoreServer per maggiori dettagli ogni qual volta si dovesse incappare in qualche errore durante il suo utilizzo.

    Ricezione dei log e dei dati delle statistiche dettagliate

    Dopo aver associato le casse, bisogna verificare se effettivamente stanno venendo trasmessi i dati e i venduti da quest'ultime allo StoreServer. Per fare ciò, è necessario controllare se tali dati stanno venendo ricevuti e immagazzinati nella tabella log del database dello StoreServer.

    Una volta accertato ciò, come ulteriore conferma, assicurarsi che i dati arrivati dalle casse stiano popolando correttamente le varie tabelle delle statistiche dettagliate della WebApp. Per consultare le statistiche dettagliate fare riferimento alla seguente guida.

    NOTA BENE: Se i record da importare nella tabella log sono numerosi, potrebbe volerci un po' di tempo prima che la situazione si allinei e tutti i dati vengano trasmessi dalle casse allo StoreServer.
    Se si hanno dei dubbi sul fatto che i dati stiano venendo effettivamente trasmessi, controllare nella cassa in cui è stato installato il nuovo Posware 4.3 il campo LastRecord della tabella config_log, che mostrerà l'ultimo record inviato da tale cassa allo StoreServer.

    Verifica del comportamento corretto dei servizi legacy

    Prima di passare all'invio massivo di tutta l'anagrafica del punto vendita allo StoreServer, è necessario verificare il comportamento corretto dei servizi legacy. I dettagli specifici sono consultabili nella rispettiva sezione.

    Invio massivo di tutta l'anagrafica del punto vendita

    NOTA BENE: Si ricorda ancora una volta che questo step conclusivo dell'installazione dello StoreServer, è obbligatorio e imprescindibile per continuare ad avere un corretto funzionamento delle operazioni del punto vendita. Essendo però un passaggio abbastanza oneroso a livello di tempistiche, prima di iniziare, assicurarsi di poter effettuare questa operazione con il punto vendita senza causare effetti collaterali indesiderati.

    Con il passaggio a Posware 4.3 e l'introduzione dello StoreServer, il vecchio database CASSAMASTER viene deprecato e quindi non sarà migrato con il Database Upgrader. Tuttavia, per questo motivo, è necessario inviare l'intera anagrafica per ripopolare le varie tabelle che componevano il database CASSAMASTER. Per eseguire questa operazione fare riferimento alla documentazione del software di backoffice in uso dal punto vendita.

    Nell'eseguire le operazioni necessarie con il software di backoffice, assicurarsi che siano rispettate le seguenti raccomandazioni:

    • Non causare stampe massive e/o superflue delle etichette prezzo.
    • Se non si ritiene necessario, evitare di inviare variazioni massive alle casse, alle bilance e ad eventuali etichette elettroniche installate. La procedura di invio massivo delle variazioni, infatti, ha il solo scopo di popolare il database del server centrale.

    Al fine di popolare il solo database dello StoreServer, NON copiare dati e strutture delle tabelle delle casse, in quanto differenti! Operazioni di questo tipo NON sono assolutamente supportate e comportano la CORRUZIONE IRRIMEDIABILE del database del server centrale!

    Rollback

    Al fine di eseguire la procedura di rollback dello StoreServer, se sono stati fatti i backup preliminari necessari, sarà necessario completare tutti i seguenti step:

    1. Disinstallare lo StoreServer, eliminando la relativa cartella dalla macchina, e ripristinare eventuali backup a database e directory non direttamente collegate a Posware, che però sono state influenzate dall'installazione dello StoreServer o di una o più dipendenze/procedure usate dalla nuova versione.
    2. Ripristinare i vari servizi legacy alla versione precedente all'aggiornamento eseguito per renderli compatibili con lo StoreServer.
    3. Ripristinare il backup del database precedente all'installazione dello StoreServer se necessario. Inoltre, se il provider del database, MySQL o SQL Server, era stato installato momentaneamente su una porta differente per non entrare in conflitto con quello precedente, procedere a disabilitarlo mettendolo offline. Infine, concludere la procedura di rollback riattivando il precedente provider se era stato disabilitato.
